## Workflow-0
This architecture follows a microservices pattern. 

- Users access a React frontend through Nginx. 
- All API requests are routed to an API Gateway, which acts as the central entry point and forwards requests to the appropriate microservice such as Auth, Product, Order, or User services. 
- Each service is independently deployable and manages its own data in PostgreSQL. 
- Monitoring is implemented using Prometheus for metrics collection and Grafana for visualization. 
- This design improves scalability, fault isolation, maintainability, and allows independent deployments of services.


- Order Service
- User Service

Each microservice is independently deployable, follows the single responsibility principle, and manages its own business logic.

The services store and retrieve data from **PostgreSQL** databases.

For observability:

- **Prometheus** collects metrics from all services.
- **Grafana** provides dashboards to visualize system health and performance.

This architecture provides:

- Scalability
- Fault Isolation
- Maintainability
- Independent Deployments
- Better Monitoring and Observability

# Interview Questions & Answers

## 1. Why did you choose Microservices instead of Monolithic Architecture?

Microservices allow:

- Independent deployment
- Independent scaling
- Easier maintenance
- Better fault isolation

If one service fails, the entire application does not necessarily go down.

## 2. What is the purpose of the API Gateway?

The API Gateway acts as a single entry point for all client requests.

Responsibilities:

- Request Routing
- Authentication
- Authorization
- Rate Limiting
- Logging
- Security

Example:

```text
/api/auth     → Auth Service
/api/products → Product Service
/api/orders   → Order Service
```

## 3. Why do we need Nginx?

### Answer

Nginx is used as a reverse proxy.

Responsibilities:

- Serve React Application
- SSL/TLS Termination
- Load Balancing
- Reverse Proxy
- Routing Requests

## 4. How does the Frontend communicate with Backend Services?

### Answer

The frontend communicates only with the API Gateway.

```text
User
 ↓
React Frontend
 ↓
Nginx
 ↓
API Gateway
 ↓
Microservices
```

This hides backend complexity from the client.

## 5. How do Microservices communicate with each other?

### Answer

Microservices can communicate using:

- REST APIs
- gRPC
- Kafka
- RabbitMQ
- Amazon SQS

The communication mechanism depends on business requirements.

## 6. Why separate Auth, Product, and Order Services?

### Answer

Each service owns a specific business capability.

Benefits:

- Loose Coupling
- Better Maintainability
- Independent Scaling
- Independent Deployments

## 7. What happens if one Microservice fails?

### Answer

Other services continue functioning.

Example:

If Product Service fails:

- User Login still works
- User Profile still works

This is called **Fault Isolation**.

## 8. How would you scale this Architecture?

### Answer

Individual services can be scaled independently.

Example:

```text
High Product Traffic
       │
       ▼
Scale Product Service Only
```

No need to scale the entire application.

For observability, Prometheus scrapes metrics from the containers, and Grafana visualizes those metrics through dashboards, helping developers monitor application health and performance.

The main advantage of this setup is that developers can run the entire microservices ecosystem locally with a single command, ensuring consistency across environments, easier testing, simplified dependency management, and faster development cycles."

### "Why Docker Compose?"

"Docker Compose helps orchestrate multiple containers locally, provides service discovery through an internal network, ensures environment consistency, and allows the complete application stack to be started or stopped using a single command."

## Workflow-2

![](/ai-ops-project/images/workflow-2.png)

"This diagram shows how cloud infrastructure is provisioned on AWS using Terraform following the Infrastructure as Code approach.

The entire infrastructure is created using a single `terraform apply` command. Terraform is organized into separate reusable modules: Networking, EKS, and ECR.

First, the Networking module creates the AWS VPC, subnets across 3 Availability Zones, Internet Gateway, and route tables. This provides a highly available network foundation for the Kubernetes cluster.

Next, the EKS module provisions the Amazon EKS cluster, worker nodes, IAM roles, and OIDC integration. The cluster uses spot instances to reduce costs and the EBS CSI driver to dynamically provision persistent storage for workloads.

The ECR module creates multiple private ECR repositories, typically one repository per microservice. These repositories store Docker images that will later be deployed to Kubernetes.

After the infrastructure is created, Terraform uses Helm to install essential Kubernetes tools such as ArgoCD for GitOps-based deployments and the kube-prometheus-stack for monitoring and observability.

Overall, this approach provides a fully automated, repeatable, and scalable cloud environment where networking, Kubernetes infrastructure, container registries, monitoring, and deployment tools are all managed through code."

### "Why Terraform?"

"Terraform allows us to provision and manage infrastructure as code, making deployments repeatable, version-controlled, automated, and consistent across environments."

### "Why EKS?"

"EKS is AWS's managed Kubernetes service. AWS manages the control plane, while we focus on deploying and scaling applications."

### "Why ArgoCD?"

"ArgoCD follows a GitOps model where Kubernetes deployments are automatically synchronized from Git repositories, providing version control, auditability, and automated rollbacks."

## Workflow-3

![](/ai-ops-project/images/workflow-3.png)

"This diagram represents the Continuous Integration (CI) pipeline implemented using GitHub Actions for a microservices-based application.

Whenever a developer pushes code to the `project-demo` branch, GitHub Actions automatically triggers the CI workflow defined in the `ci.yml` file.

Since the application consists of multiple microservices such as Auth, Product, Orders, Order Management, User Service, API Gateway, and Frontend, the pipeline uses a matrix build strategy to build all services in parallel. This significantly reduces the overall build time compared to building each service sequentially.

For each service, the pipeline first authenticates with Amazon ECR using AWS credentials. It then builds a Docker image, tags the image using the Git commit SHA to ensure unique and traceable versions, and pushes the image to its corresponding private ECR repository.

Using commit SHA tags helps us identify exactly which source code version is running in production and makes rollbacks easier if any issue occurs.

Once the pipeline completes successfully, all updated Docker images are available in ECR and ready to be deployed to the EKS cluster through the deployment pipeline, typically using GitOps tools like ArgoCD.

This approach provides automation, faster build times, version control, traceability, and a reliable software delivery process."

### "Why use Matrix Builds?"

"Matrix builds allow multiple microservices to be built simultaneously, reducing overall pipeline execution time compared to sequential builds."

### "Why push images to ECR?"

"ECR serves as a secure private container registry where Kubernetes can pull versioned images for deployment."

### "Why use Commit SHA as Image Tag?"

"Commit SHA provides unique and immutable image versions, enabling traceability, auditing, and easy rollback to previous releases."

### "What is the difference between CI and CD?"

**CI (Continuous Integration):**

- Build code
- Run tests
- Create Docker images
- Push artifacts

**CD (Continuous Delivery/Deployment):**

- Deploy images to Kubernetes
- Update environments automatically
- Manage releases and rollbacks

In this diagram, we are primarily looking at the **CI stage**.

"This diagram shows the observability setup for a Kubernetes-based microservices application using Prometheus and Grafana.

Each microservice, such as Auth, Product, Orders, Order Management, and User Service, exposes a /metrics endpoint. These endpoints publish application metrics like request count, response time, error rates, CPU usage, and memory consumption in a format that Prometheus can understand.

In Kubernetes, a ServiceMonitor resource is used to tell Prometheus which services and pods should be monitored. Instead of manually configuring targets, Prometheus automatically discovers the services through ServiceMonitor.

Prometheus periodically scrapes these metrics endpoints, typically every 15 seconds, and stores the collected data as time-series metrics.

Grafana is connected to Prometheus as a data source and provides dashboards to visualize application and infrastructure health. Using Grafana, teams can monitor API latency, request throughput, pod health, resource utilization, and error rates in real time.

This observability stack helps quickly identify performance bottlenecks, troubleshoot issues, monitor application health, and create alerts for proactive incident management."

## Workflow-5

![](/ai-ops-project/images/workflow-5.png)

"This workflow demonstrates the Continuous Deployment (CD) process using ArgoCD and the GitOps methodology.

After the CI pipeline builds and pushes a new Docker image to Amazon ECR, the developer updates the image tag in the Kubernetes manifest files stored in the GitOps repository. The changes are then committed and pushed to Git.

ArgoCD continuously monitors the Git repository and compares the desired state defined in Git with the actual state running in the Kubernetes cluster. When ArgoCD detects a change, such as a new image tag, it automatically synchronizes the cluster by applying the updated Kubernetes manifests.

ArgoCD is configured with features like Auto Sync, Self-Heal, and Prune. Auto Sync automatically deploys changes, Self-Heal ensures the cluster matches the Git repository even if manual changes occur, and Prune removes resources that no longer exist in Git.

Once the updated manifests are applied, Kubernetes performs a rolling update. New pods are created with the latest image version, traffic is gradually shifted to the new pods, and old pods are terminated only after the new ones become healthy. This ensures zero or minimal downtime during deployments.

This GitOps approach improves deployment reliability, auditability, rollback capabilities, and ensures Git remains the single source of truth for the entire Kubernetes environment."

### "What is GitOps?"

"GitOps is a deployment methodology where Git acts as the single source of truth. Any infrastructure or application changes are made through Git, and deployment tools automatically synchronize those changes to Kubernetes."

### "What is ArgoCD?"

"ArgoCD is a GitOps continuous delivery tool for Kubernetes that automatically deploys applications by synchronizing Kubernetes manifests stored in Git repositories."

### "How does ArgoCD detect changes?"

"ArgoCD continuously watches the Git repository. When it detects a change in manifests, image tags, Helm charts, or Kustomize configurations, it automatically triggers synchronization."

### "What is Auto Sync in ArgoCD?"

"Auto Sync automatically applies changes from Git to Kubernetes without requiring manual deployment actions."

### "What is Self-Heal?"

  "Self-Heal automatically restores resources if someone manually modifies or deletes them in the Kubernetes cluster, ensuring the cluster always matches the Git repository."

### "What is Prune?"

"Prune removes Kubernetes resources that have been deleted from the Git repository, preventing orphaned resources from remaining in the cluster."

### "What is a Rolling Update?"

"A rolling update gradually replaces old pods with new ones without bringing down the entire application, minimizing downtime and deployment risk."

### "What are the benefits of GitOps?"

- Version-controlled deployments
- Easy rollback
- Auditability
- Automation
- Improved security
- Consistent environments

### "How do you roll back a deployment in GitOps?"

Rollback is achieved by reverting the Git commit containing the problematic change. ArgoCD detects the revert and automatically synchronizes the cluster back to the previous stable version.

### "Why use ArgoCD instead of running kubectl apply manually?"

"ArgoCD provides automated deployments, drift detection, self-healing, version control integration, audit trails, and eliminates manual deployment errors."
